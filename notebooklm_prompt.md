이 노트북에는 GDC(Game Developers Conference) 발표 하나를 분석하기 위한 여러 소스 파일이 업로드되어 있습니다. 각 파일의 성격이 다르므로 아래 설명을 참고하여 교차 인용하며 답변해 주세요.

### 업로드된 소스 (예상 구성)

- `meta.md` — 세션 제목, 발표자/소속, GDC 연도, 트랙, 태그, 공식 Overview. **고유명사의 캐노니컬 철자 기준**으로 삼을 것.
- `transcript.txt` — 발표 전체 본문. 상단에 메타 헤더가 붙어 있음. ASR 원본이라 일부 고유명사가 음성 인식 오류로 남아 있을 수 있으나, `meta.md` 와 `glossary.md` 를 기준으로 해석할 것.
- `transcript_timed.txt` — 타임스탬프가 포함된 본문. 특정 발언의 시점 인용이 필요할 때 참조.
- `chapters.md` — AI가 정리한 챕터 구조 (타임스탬프 + 한 줄 요약). 발표 전체 흐름 파악용.
- `keypoints.md` — 발표자의 3~5개 핵심 주장과 각 주장의 구체적 증거(수치, 사례, 일화). 슬라이드 본문의 기둥.
- `glossary.md` — 언급된 게임 타이틀, 스튜디오, 인물, 전문 용어의 정의집. 인포그래픽/용어 설명에 활용.
- `qa.md` *(있을 경우)* — 발표 후 청중 Q&A. 실무자들이 실제로 궁금해하는 지점이 드러나므로 **FAQ 슬라이드**와 멘토링 포인트에 적극 활용.
- `design_brief.md` *(있을 경우)* — 이 세션에서 다루는 게임의 비주얼 아이덴티티 요약 (아트 스타일, 컬러 팔레트 hex 코드, 타이포그래피, 시그니처 모티프, 회피해야 할 트로프). **슬라이드 덱의 전체 비주얼 테마는 반드시 이 파일을 따를 것.**
- `related_articles.md` *(있을 경우)* — 해외 게임 언론(gamedeveloper.com 등)에 보도된 관련 기사. **업계 수치·평가·후속 뉴스**를 인용할 때 출처로 사용.
- `thumbnail.jpg` *(있을 경우)* — 표지 슬라이드의 시각 자료로 참조 가능.

### [슬라이드 덱 생성]

- **비주얼 테마 (컬러·타이포는 살리고, 아이코닉 모티프는 절제)**: `design_brief.md` 가 있으면 다음 원칙으로 적용한다.
  - **적극 활용 — Color Palette & Typography**: 덱 전체(표지·섹션·본문 모두)의 컬러 톤과 서체는 그 게임의 아이덴티티를 따라도 좋다. 본문 슬라이드의 배경색·헤딩·강조·하이라이트·인용 바·차트 액센트에 팔레트 컬러를 자연스럽게 사용. 디스플레이 폰트는 제목/섹션, 본문 폰트는 가독성 우선 서체로 분리.
  - **매우 절제해서 활용 — Signature Motifs**: 게임의 상징적 오브젝트(예: 톱니바퀴·총·검·카드 문양·특정 캐릭터 실루엣·시그니처 프롭 등)는 **덱 전체에서 1~2종류만** 고르고, **표지 슬라이드와 섹션 구분 슬라이드의 작은 장식, 또는 본문 페이지 번호 옆의 미세한 글리프** 정도로만 사용. 각 모티프는 `design_brief.md` 의 Signature Motifs 블록에 적힌 scope(`cover accent`, `section-divider icon`, `corner decoration`, `one-time accent`)를 반드시 준수할 것.
  - **금지 (집중 저해)**: 같은 아이코닉 심볼(톱니바퀴·총·검 등)을 여러 슬라이드에 반복 배치. 하나의 모티프를 반복 패턴(타일링·배경 무늬)으로 사용. 본문 슬라이드 중앙이나 배경에 심볼을 크게 배치. 모든 불릿 포인트 앞에 테마 아이콘 장식. 본문 텍스트·차트 뒤에 풀블리드 게임 아트를 깔기.
  - **Avoid 섹션 존중**: `design_brief.md` 의 Avoid 항목은 그대로 반영.
  - `design_brief.md` 가 없을 때만 아래 "세련된 게임 스튜디오 톤"을 기본값으로 사용.
- **가독성 우선**: 본문 슬라이드는 텍스트·도표·수치가 한눈에 읽히는 것이 목적. 테마의 일관성(특히 심볼 반복)보다 정보 전달이 우선.
- **디자인 및 시각화**: 복잡한 시스템·아키텍처·수치는 단순 텍스트 나열을 피하고 인포그래픽이나 시각적 은유(Metaphor)를 활용하여 시각화. 은유는 게임의 모티프를 참고하되 위의 모티프 제약(1~2종, 소형, 특정 위치) 안에서만 사용.
- **고유명사 일관성**: 게임 타이틀·스튜디오·인물명은 `meta.md` 와 `glossary.md` 의 철자를 그대로 사용. `transcript.txt` 에 남아있을 수 있는 ASR 오인식 철자는 반영하지 말 것.
- **수치 인용**: `keypoints.md` 의 증거와 `related_articles.md` 의 업계 지표를 우선 인용하고, 출처를 괄호로 명시 (예: *"5백만 장 판매 (gamedeveloper.com, 2025)"*).
- **슬라이드 구성** (각 장마다 제목, 1문장 핵심 메시지, 3~5개 불릿 포인트 포함):
  1. **표지**: 발표 주제, 발표자/소속, GDC 연도 — `meta.md` 기반. `thumbnail.jpg` 가 있으면 배경으로 활용.
  2. **문제 정의**: 이 발표가 해결하려는 핵심 과제 — `keypoints.md` 의 맥락에서 추출.
  3. **발표 로드맵**: `chapters.md` 를 압축하여 5~7단계 흐름으로 시각화.
  4~11. **본문**: `keypoints.md` 의 각 Key Point를 한두 장씩 할애. 제안하는 해결 방법, 언급된 게임 타이틀과 사례(`glossary.md` 참조), 구체적 수치 강조.
  12. **업계 맥락 / 성과**: `related_articles.md` 의 보도를 활용해 이 발표의 의의, 관련 제품의 상업적 성과, 후속 뉴스를 요약.
  13. **청중이 궁금해한 것 (FAQ)**: `qa.md` 가 있으면 가장 실무적인 질문 3~5개와 발표자의 답변 핵심을 슬라이드로 구성. 없으면 생략하고 본문을 1장 확장.
  14. **핵심 교훈 (Lessons Learned)**: 발표자의 관점 + Q&A에서 드러난 실무적 뉘앙스를 결합.
